# 情侣日常 App 开发计划文档

## 1. 项目概述与目标

### 1.1 项目背景
情侣日常 App 面向异地或日常需要高频互动的情侣用户，核心价值在于“连接、记录、提醒、共创”。产品通过聊天、共享相册、纪念日、心情打卡、位置共享、日记与愿望清单等能力，帮助用户建立持续的情感互动闭环，提高留存与使用频率。

### 1.2 项目目标
- 搭建一套可持续演进的 Flutter 跨平台客户端架构，支持 iOS 与 Android 双端发布。
- 在首个正式版本内完成情侣关系绑定、互动沟通、生活记录、重要日子提醒四类核心场景覆盖。
- 优先保证核心链路稳定：登录注册 -> 情侣绑定 -> 首页 -> 聊天/相册/纪念日/心情打卡。
- 通过分阶段开发，控制交付风险，确保 MVP 尽快上线验证用户反馈。
- 为后续扩展实时位置共享、日记共创、愿望清单、推送通知等能力预留架构空间。

### 1.3 产品范围
#### MVP 范围
- 登录注册
- 情侣绑定
- 首页聚合
- 聊天
- 共享相册
- 纪念日
- 心情打卡

#### 增强范围
- 位置共享
- 双人日记
- 愿望清单
- 推送通知

### 1.4 成功指标
- 首次安装到完成情侣绑定转化率 >= 60%
- 核心功能崩溃率 < 1%
- 聊天消息成功到达率 >= 99%
- 纪念日提醒触达成功率 >= 95%
- 用户 7 日留存达到 25% 以上

### 2.3 后端与云服务建议
| 模块 | 推荐方案 | 说明 |
|------|----------|------|
| 用户认证 | Firebase Authentication / 自建 JWT 服务 | 快速接入手机号、邮箱认证 |
| 实时聊天 | WebSocket + Socket.IO / Firebase Firestore | 需要保障消息实时性 |
| 文件存储 | Firebase Storage / 阿里云 OSS / 七牛云 | 用于图片、相册、日记附件 |
| 数据库 | Firestore / Supabase / 自建 MySQL + NestJS | 根据团队后端能力决定 |
| 推送服务 | Firebase Cloud Messaging | 跨平台统一推送能力 |
| 地图服务 | 高德地图 Flutter SDK | 位置共享与地理围栏能力 |

### 2.4 架构建议
- 架构模式：`Presentation + Domain + Data` 三层结构。
- 页面组织：按业务模块拆分目录，避免按类型平铺导致耦合过高。
- 状态策略：
  - 页面级轻交互：GetX Controller。
  - 核心业务流：BLoC/Cubit。
- 网络层：统一封装 `ApiClient`、错误码处理、Token 刷新逻辑。
- 存储层：敏感数据走 `flutter_secure_storage`，普通缓存走 `Hive`。
- 可测试性：Repository 抽象接口，核心 UseCase 可单测覆盖。

## 3. 开发阶段划分

## Phase 1：项目基础
### 阶段目标
完成项目工程初始化、基础架构搭建、通用能力沉淀，为后续业务开发建立统一标准。

### 交付物
- Flutter 工程初始化与目录规范
- 路由系统与页面跳转框架
- 网络层、异常处理、通用响应模型
- 状态管理基础设施
- 主题系统（颜色、字体、暗黑模式预留）
- 基础组件库（按钮、输入框、空状态、加载态、对话框）
- 环境配置（dev/test/prod）
- CI 基础脚本与静态检查配置

### 预计工作量
- 8 ~ 10 人日
- 1 名前端主开发 + 1 名兼职后端/架构支持

## Phase 2：核心功能
### 阶段目标
打通用户从注册登录到建立情侣关系，再到进入主页使用核心功能的主流程。

### 交付物
- 登录/注册/找回账户流程
- 情侣邀请码/二维码绑定流程
- 首页聚合页
- 用户资料与情侣资料基础信息管理
- Token 管理、会话保持、登录态恢复
- 核心埋点方案第一版

### 预计工作量
- 12 ~ 15 人日
- 1 名前端主开发 + 1 名后端开发 + 1 名测试兼职支持

## Phase 3：业务功能
### 阶段目标
完成 MVP 核心业务能力，支持情侣高频使用场景，具备小范围内测条件。

### 交付物
- 聊天模块
- 共享相册模块
- 纪念日模块
- 心情打卡模块
- 主页快捷入口与聚合数据联动
- 基础消息提醒能力
- 内测版本打包与问题修复

### 预计工作量
- 18 ~ 22 人日
- 2 名前端开发 + 1 名后端开发 + 1 名测试

## Phase 4：进阶功能
### 阶段目标
补齐产品差异化能力，提升用户粘性、仪式感与长期留存表现，推动首个正式版发布。

### 交付物
- 实时位置共享
- 双人日记
- 愿望清单
- 推送通知完整链路
- 设置中心增强项
- 性能优化与发布前安全检查
- 应用商店发布包

### 预计工作量
- 15 ~ 18 人日
- 2 名前端开发 + 1 名后端开发 + 1 名测试 + 1 名产品验收

## 6. 风险评估与应对

| 风险项 | 风险描述 | 影响 | 应对措施 |
|--------|----------|------|----------|
| 架构选型不稳 | 初期状态管理与模块边界不清，导致后期重构成本高 | 高 | Phase 1 完成架构评审，核心模块先抽象接口 |
| 实时聊天复杂度高 | 消息同步、离线重连、未读状态处理复杂 | 高 | 优先打通最小消息链路，复杂能力分步实现 |
| 位置共享隐私敏感 | 定位权限、后台定位、用户隐私合规要求高 | 高 | 默认关闭、强提醒授权、双向确认、最小化存储 |
| 多媒体上传不稳定 | 弱网环境下上传失败或大图性能差 | 中 | 接入压缩、断点重试、失败重传机制 |
| 推送到达率不稳定 | 厂商限制导致通知无法稳定到达 | 中 | 使用 FCM + 本地通知兜底，重要提醒支持站内二次提醒 |
| 后端接口变动 | 接口设计未冻结导致客户端返工 | 中 | 接口文档前置、Mock 数据先行、每日同步机制 |
| 工期压缩 | 资源有限导致功能过多难以按期交付 | 高 | 明确 MVP 边界，优先核心场景，进阶功能可降级上线 |
| 测试覆盖不足 | 新功能多且节奏快，容易带入回归问题 | 中 | 核心流程建立冒烟清单，版本发布前固定回归模板 |

### 6.1 风险优先级处理原则
- 高风险项必须在对应阶段开始前完成预案确认。
- 所有涉及隐私、定位、消息数据的问题优先级高于视觉优化。
- 新增需求必须经过“影响评估 -> 工期评估 -> 版本取舍”后才能进入开发。

## 7. 开发规范约定

### 7.1 Git Flow 约定
采用简化版 Git Flow：
- `main`：正式发布分支，只允许合并发布版本代码。
- `develop`：日常开发主分支，所有功能完成后合并到该分支。
- `feature/*`：功能开发分支，如 `feature/login`、`feature/chat-room`。
- `release/*`：发布预发布分支，如 `release/v1.0.0`。
- `hotfix/*`：线上紧急修复分支，如 `hotfix/push-fix`。

分支流程：
1. 从 `develop` 拉出 `feature/*`
2. 功能自测通过后发起 PR 合并到 `develop`
3. 阶段版本冻结时从 `develop` 拉出 `release/*`
4. 验证通过后合并到 `main` 与 `develop`
5. 线上紧急问题从 `main` 拉 `hotfix/*` 修复

### 7.2 代码规范
- 使用 `flutter_lints` 作为基础规范，提交前执行 `dart format` 与 `flutter analyze`。
- 文件命名统一使用小写下划线，例如 `chat_message_item.dart`。
- 类名使用大驼峰，变量和方法使用小驼峰。
- 页面、控制器、仓库、模型按模块分目录管理。
- 每个业务模块最少包含：`page`、`controller/bloc`、`repository`、`model`。
- 禁止页面直接发起原始网络请求，必须通过 Repository 或 Service 层访问。
- 禁止在 Widget 中堆积业务逻辑，复杂逻辑下沉到 Controller/BLoC/UseCase。
- 敏感信息不得硬编码到仓库，统一放在环境变量或安全配置中。

### 7.3 Commit Message 格式
采用 Conventional Commits：

```bash
<type>(scope): <subject>
```

常用类型：
- `feat`：新功能
- `fix`：缺陷修复
- `refactor`：重构
- `style`：格式调整，不影响逻辑
- `docs`：文档更新
- `test`：测试相关
- `chore`：构建、脚本、依赖更新

示例：
```bash
feat(auth): 完成手机号登录流程
feat(chat): 新增图片消息发送能力
fix(album): 修复相册上传失败重试问题
docs(plan): 补充项目开发计划文档
```

### 7.4 PR 规范
- PR 标题与 Commit Message 风格保持一致。
- PR 描述必须包含：变更内容、影响范围、测试结果、回滚方案。
- UI 变更必须附带截图或录屏。
- 核心功能 PR 至少 1 人 Review 后才能合并。
- 合并前必须确保无分析告警、无冲突、基础用例通过。

### 7.5 测试约定
- 核心流程必须覆盖冒烟测试：登录、绑定、发送消息、上传图片、添加纪念日、心情打卡。
- 核心业务模块建议覆盖单元测试：Repository、UseCase、消息状态处理。
- 页面级至少覆盖关键交互 Widget Test。
- 每个版本发布前必须执行一次完整回归测试。

## 8. 资源建议与协作方式

### 8.1 角色建议
- 产品经理：负责需求冻结、原型确认、验收标准输出。
- Flutter 开发：负责客户端实现与性能优化。
- 后端开发：负责认证、聊天、文件、推送与业务接口。
- 测试：负责冒烟、回归、兼容性与发布验证。
- UI 设计：负责视觉规范、组件态与动效补充。

### 8.2 协作机制
- 每日站会同步进度、阻塞与风险。
- 每周固定评审一次版本目标与燃尽情况。
- 所有需求变更通过文档记录，并同步工期影响。
- 接口变更必须同步更新接口文档与 Mock 数据。

---

## 9. 建议的开发优先级结论
1. 优先保证登录注册、情侣绑定、首页聚合、聊天链路可用。
2. 相册、纪念日、心情打卡作为 MVP 差异化能力同步推进。
3. 位置共享与推送通知属于高价值高风险功能，放在后期集中攻坚。
4. 双人日记与愿望清单用于增强留存，可在 MVP 稳定后快速补齐。

## 10. 文档结论
本计划以 4 个阶段推进，目标是在 `2026-07-04` 发布 `v1.0.0` 正式版。执行过程中应严格控制需求范围，确保先完成核心链路，再扩展进阶功能，以降低返工风险并提升首版上线成功率。
